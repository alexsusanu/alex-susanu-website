IMMEDIATE RESPONSE (0-30 min)

1. Acknowledge & Get Help FAST
Scenario: SEV1 Payment API outage - You're new (2 weeks on job) → PagerDuty: Click "Acknowledge" (you know this much) → Slack: Immediately post "#incidents HELP - Payment SEV1, I'm new, need backup" → Escalate NOW: Don't try to investigate alone - hit "Escalate" in PagerDuty → Find your buddy: Slack DM the senior engineer assigned as your mentor
Why: Being new during SEV1 = ask for help immediately. Don't hero it and make it worse
2. Get Context While Senior Takes Lead
Scenario: Senior engineer responds and takes technical lead → You: "Where do I find the payment dashboard?" → Senior: "Datadog bookmark folder 'Critical Dashboards'" → You observe: New Relic APM shows payment service 100% error rate → Ask: "Should I check the runbook?" Senior: "Yes, Confluence, search 'payment outage'"
Why: Learn the tools/locations while someone experienced handles the critical path
3. IC Assigns You Supporting Role
Scenario: Engineering Manager becomes IC, gives you safe tasks → IC: "You monitor the dashboards and report any changes" → IC: "Senior engineer will execute runbook, you take notes" → Your job: Watch Datadog payment dashboard, call out if numbers change → Take detailed notes: What buttons senior engineer clicks, what works/doesn't
Why: IC gives new person low-risk but valuable tasks. You learn without breaking things
4. Learn the Team & Tools in Real-Time
Scenario: Observe who gets called and why → Watch: IC pages "Payments Team - SEV1" policy → Ask senior: "Who's Sarah and why did we page her?" "She knows Stripe integration" → Learn: Customer Support Manager gets notified for SEV1 user impact → Observe: How StatusPage.io gets updated, who has access
Why: Real incident shows you the team structure and tool usage under pressure
5. Support War Room Setup
Scenario: Help with coordination while learning → IC creates #sev1-payments-20250528, adds you as observer → Your task: Pin important links/info to channel as IC directs → Learn: How bridge calls work, who talks vs who listens → Note: StatusPage.io auto-updates, Jira ticket auto-created
Why: Supporting tasks let you see the full incident process without critical responsibility

SHORT-TERM STABILIZATION (30-60 min)

6. OBSERVE Runbook Execution
Scenario: Senior engineer executes, you watch and learn → Senior: "New person, watch me do this. Go to Confluence payment runbook" → You navigate: /wiki/runbooks/payment-system-outage (bookmark this!) → Senior executes: Switch to PayPal processor, you watch the admin dashboard → You ask: "Why PayPal and not the other one?" Learn: "PayPal handles our volume"
Why: Don't touch production during SEV1 as new person. Learn the process safely
7. Help with Investigation (Safe Tasks)
Scenario: Senior gives you investigation tasks that can't break things → Senior: "Check New Relic logs for 'payment error' in last 30 minutes" → You find: "Connection timeout to stripe.com" - report this finding → Senior: "Good, now check Stripe status page" - you find they have an outage → You document: Timeline of when their outage started vs our alerts
Why: Investigation tasks are safe for new people. You contribute without risk
8. Monitor & Learn Dashboard Locations
Scenario: Learn where everything is while monitoring emergency fixes → Senior shows you: "Datadog payments dashboard is in 'Critical' folder" → You monitor: PayPal processor success rate (now 60% of normal volume) → Learn: PagerDuty has "Suppress alerts" feature for known issues → Document: Where to find payment dashboards, what metrics matter
Why: Real incident teaches you tool locations faster than any training
9. Support Communication
Scenario: Help with updates while learning communication flow → Senior asks: "Update the incident channel with PayPal status" → You post: "PayPal processor active, 60% payment volume restored" → Learn: IC handles executive communication, technical team handles facts → Observe: How StatusPage.io updates are written for customers
Why: Communication tasks are safe and show you incident management process
10. Document Everything You Learn
Scenario: Turn this incident into learning opportunity → Your notes: "Payment runbook at /wiki/runbooks/payment-system-outage" → Your notes: "Datadog payment dashboard in Critical folder" → Your notes: "Always check external service status pages first" → Ask senior: "What should I have done differently?" Get feedback
Why: New person should treat every incident as intensive learning experience

LONG-TERM RESOLUTION (60+ min)

11. Implement Proper Fix
Scenario: Root cause identified as inefficient database query → Rewrite query to use proper indexes → Add query timeout (5 seconds) to prevent hanging → Implement caching layer for frequently accessed data → Test in staging environment first
Why: Permanent solutions prevent recurrence. Staging testing reduces deployment risk
12. Gradual Restoration
Scenario: Canary deployment approach → 10% of traffic to fixed system (monitor for 15 minutes) → If stable: 25% → 50% → 75% → 100% → Watch: checkout success rate, database CPU, error logs → Rollback plan ready if issues detected
Why: Gradual rollout catches edge cases before they affect all users
13. Verify Full Recovery
Scenario: Multi-point verification → Technical metrics: 99.5% checkout success rate (baseline) → User impact: Customer support ticket volume back to normal → Business metrics: Revenue per hour restored → Social listening: No complaints on Twitter/Reddit
Why: Technical metrics can be misleading. User experience is the real test
14. Official Stand Down
Scenario: Clear handoff and closure → IC announces: "Incident officially resolved" → Database team monitors for next 24 hours → On-call rotation resumes normal schedule → Follow-up tasks tracked in JIRA with owners and deadlines
Why: Prevents confusion about ongoing responsibilities. Ensures follow-up doesn't get forgotten

FOLLOW-UP (24-48 hours)

15. Blameless Post-Mortem
Scenario: Learning-focused retrospective within 48 hours → Attendees: All incident responders + key stakeholders → Timeline reconstruction: What happened when → 5 Whys analysis: Why did the inefficient query get deployed? → Contributing factors: Missing load testing, no query review process → Action items: Specific, measurable, with owners and deadlines
Why: Blame discourages honesty. Focus on system improvements prevents recurrence
16. Actually Do Improvements
Scenario: Execute post-mortem action items → Add database query performance monitoring (alerts on >2s queries) → Implement mandatory query review for database changes → Increase connection pool size from 50 to 100 → Add load testing to CI/CD pipeline for all database changes
Why: Most incidents repeat if underlying conditions aren't addressed
17. Update Runbooks
Scenario: Improve incident response documentation → Add "Checkout Down" runbook with specific troubleshooting steps → Update database troubleshooting: "Check slow query log first" → Document new monitoring thresholds and escalation triggers → Include architecture diagrams showing dependencies
Why: Next responder shouldn't rediscover troubleshooting steps during crisis
18. Share Lessons Learned
Scenario: Engineering-wide knowledge sharing → Email summary: "Key lesson: Load test all database schema changes" → Include: What failed, why it failed, how we fixed it, how to prevent → Engineering all-hands presentation on incident response best practices → Update onboarding materials with new procedures
Why: Prevents other teams from making similar mistakes. Builds organizational resilience

FIRST 30 DAYS - Foundation Building

Week 1-2: Tool Discovery & Access
• Get access to all monitoring tools (Datadog, New Relic, PagerDuty) • Bookmark critical dashboards in "30-Day Learning" folder • Set up Slack notifications for #incidents channel • Complete security training for production access
Goal: Know where everything is located when crisis hits
Week 2-3: Shadow & Document
• Shadow every incident (even if 3AM) - take detailed notes • Document: Who gets called for what type of issue • Create personal cheat sheet: "Payment issue = page Sarah, DB issue = page Mike" • Practice using runbooks on staging environment
Goal: Understand team dynamics and escalation patterns
Week 3-4: SEV3 Practice
• Take ownership of SEV3 incidents with senior engineer backup • Practice: "I acknowledge, investigate for 15 minutes, then escalate if stuck" • Learn: How to write good incident updates in Slack • Study: Past incident post-mortems for common failure patterns
Goal: Build confidence with low-risk incidents
End of Month Assessment
• Can you find payment dashboard in under 30 seconds? • Do you know when to escalate vs investigate further? • Can you execute basic runbook steps without help? • Ready for SEV3 on-call shifts with senior backup
Goal: Self-sufficiency for minor incidents

30-60 DAYS - Building Competence

Week 5-6: SEV3 Leadership
• Lead SEV3 incidents from start to finish independently • Practice post-incident communication: "What happened, what we did, follow-up" • Learn advanced log searching in Splunk/New Relic • Start contributing to runbook improvements based on your experience
Goal: Independent handling of minor incidents
Week 6-7: SEV2 Shadowing
• Shadow SEV2 incidents, take on safe investigation tasks • Practice: "Check external service status pages while senior handles runbook" • Learn: Database query basics for payment/user lookup during incidents • Understand: When to suggest rollbacks vs fixes forward
Goal: Useful contributor to medium severity incidents
Week 7-8: System Architecture Deep Dive
• Study system architecture diagrams in Confluence • Understand: Payment flow from frontend → API → database → Stripe • Learn: Common failure points and their symptoms • Practice: Explaining technical issues to non-technical stakeholders
Goal: Understand system dependencies for better troubleshooting
60-Day Assessment
• Can you lead SEV3 incidents confidently? • Do you understand payment/user/order system architecture? • Can you assist meaningfully in SEV2 incidents? • Ready to be backup on-call for evenings/weekends
Goal: Trusted team member for most incident types

60-90 DAYS - Full Capability

Week 9-10: SEV2 Leadership
• Lead SEV2 incidents with senior engineer as backup (not lead) • Practice: Coordinating multiple people during multi-system failures • Learn: When to make vs escalate business impact decisions • Master: All common runbooks without constantly referring to documentation
Goal: Independent handling of significant incidents
Week 10-11: SEV1 Contributor
• Contribute as technical responder in SEV1 incidents • Practice: Rapid parallel investigation while senior handles critical path • Learn: How to communicate technical findings to Incident Commander clearly • Understand: Business impact calculation and revenue implications
Goal: Valuable contributor to critical incidents
Week 11-12: Advanced Troubleshooting
• Troubleshoot issues without relying on runbooks • Practice: Creating new runbook steps for novel issues • Learn: Performance analysis, database optimization basics • Master: Cross-team coordination (DevOps, Product, Customer Support)
Goal: Handle new types of incidents independently
90-Day Assessment & Full On-Call
• Can you handle SEV2 incidents independently? • Are you a useful contributor to SEV1 incidents? • Do you troubleshoot systematically without step-by-step guidance? • Ready for full on-call rotation including primary responder shifts
Goal: Full team member ready for independent on-call duty

WHEN THINGS GO WRONG

Runbook Doesn't Work
Scenario: You follow payment outage runbook, but backup processor also fails STOP: Don't keep trying broken steps Slack incident channel: "Runbook step 2 failed - PayPal also down" Escalate IMMEDIATELY: "Need senior engineer - runbook not working" Document: Which steps failed and exact error messages
Don't waste time on broken procedures. Escalate fast with specific failure details
Your Emergency Fix Makes It Worse
Scenario: You enable maintenance mode but it breaks the whole site REVERT IMMEDIATELY: Undo your last change first Slack: "Maintenance mode broke site - reverted - need help" Don't hide it: Tell IC exactly what you did and when Learn: Always test emergency procedures on staging first
Own mistakes fast. Hiding problems during SEV1 makes everything worse
Multiple Systems Failing (Cascade Failure)
Scenario: Payment down, then database alerts, then login system fails Don't chase every alert: Focus on PRIMARY business impact Tell IC: "Multiple systems failing - need to prioritize" IC decides: "Fix payments first, ignore login for now" Suppress non-critical alerts: Reduce noise, focus on core systems
During cascades, you can't fix everything. IC must prioritize business impact
Can't Reach Key People
Scenario: Payment expert not responding, database expert on vacation PagerDuty: Escalate to next person in policy (don't wait) Check team docs: "Who's backup for Sarah?" Slack: Post in team channels "#payments-team URGENT need help" Last resort: Wake up engineering director for emergency contact info
Have backup plans for backup plans. Someone always knows someone who can help

PRESSURE SITUATIONS

CEO Demanding Constant Updates
Scenario: CEO joins incident channel asking for updates every 2 minutes
• IC handles this: "CEO, updates every 15 minutes in email thread"
• If you're IC: Set boundaries: "Technical team needs focus time"
• Designate communicator: Someone shields technical team from executive pressure
• Stick to schedule: "Next update at 3:45 PM as promised"
Executive pressure kills technical focus. Set boundaries and stick to communication schedule
Social Media / Customer Meltdown
Scenario: Twitter exploding with complaints, customers calling support non-stop
• Your job: Focus on technical fix, ignore social media noise
• Customer support: Handles external communication with your updates
• StatusPage.io: Keep external updates honest but not too technical
• Don't read Twitter: It will stress you out and slow down fix
Customer anger is motivating but distracting. Let support handle external communication
Revenue Impact Pressure
Scenario: "$100k/hour lost" dashboard ticking up in real-time
• Remember: Pressure to go fast often causes more mistakes
• IC reminds team: "Fix it right, not fast"
• Take calculated risks: Don't skip testing just because of pressure
• Communicate progress: "60% restored" reduces pressure on team
Revenue pressure makes people take shortcuts. Discipline prevents bigger disasters
How to Stay Calm Under Pressure
Techniques for high-stress incidents:
• Breathe deliberately: 4 seconds in, 4 seconds out
• Focus on next step only: Don't think about the whole problem
• Use your training: Follow procedures, don't improvise under stress
• Communicate frequently: Talking reduces anxiety for everyone
Calm teams make better decisions. Stress makes people skip steps and make mistakes

REAL WORLD CHAOS

Documentation is Wrong/Outdated
Scenario: Runbook says "click button X" but button doesn't exist
• Don't waste time: Try for 2 minutes max, then escalate
• Slack: "Step 3 in runbook is wrong - button missing - need help"
• Senior engineer: "Oh yeah, we moved that to the new admin panel"
• Fix it later: Update runbook after incident is resolved
Bad docs are common. Don't assume you're doing it wrong - escalate quickly
Monitoring Tools Are Down
Scenario: Datadog is down during payment outage - no dashboards
• Backup plan: Use New Relic, Grafana, or raw server logs
• Manual checks: SSH to servers, check application logs directly
• Customer reports: Sometimes users tell you what's broken
• External monitoring: Use Pingdom, UptimeRobot if available
When tools fail, go to basics. Multiple monitoring sources prevent blindness
Key Expert on Vacation/Unreachable
Scenario: Payment system architect is on vacation in remote area
• Check team knowledge base: "Payment system overview" docs
• Find apprentice: Who was learning from the expert?
• Vendor support: Call Stripe directly if it's their issue
• Historical incidents: Look at past payment outages and solutions
Institutional knowledge shouldn't depend on one person. Multiple information sources help
Everything's on Fire (Multiple SEV1s)
Scenario: Payment down, database corrupted, AWS region outage
• IC triages: "What kills the business fastest?"
• Split teams: Payment team, database team, infrastructure team
• Sequential fixes: Don't try to fix everything at once
• Accept some damage: Focus on the biggest business impact first
Perfect storms happen. Triage ruthlessly and accept you can't save everything
When You Don't Know What You Don't Know
Scenario: System behaving weird but you can't figure out why
• Describe symptoms exactly: "Payment API returns 200 but no database writes"
• Get help fast: Don't spend 30+ minutes confused alone
• Share screen: Sometimes another pair of eyes sees the issue
• Revert recent changes: Go back to known good state first
Unknown unknowns are the hardest. Get help and revert to safety quickly

TECHNICAL GOTCHAS

Rollback Also Fails
Scenario: Payment outage, you try to rollback deployment but rollback breaks database STOP rolling back: Don't make it worse Alert IC immediately: "Rollback failed - database errors now" Emergency mode: Find the last known working deployment (2-3 versions back) Manual intervention: May need database team to restore from backup Document exactly: What happened during rollback attempt
Rollbacks aren't magic - they can fail too. Sometimes you have to go further back
"Known Good State" Was Never Actually Good
Scenario: You rollback to "working" version but customers report it was already broken Realize the truth: Current issue exposed existing hidden problem Don't keep rolling back: You'll just find more broken versions Focus forward: Fix the underlying issue instead of seeking perfect past Communicate honestly: "Rollback revealed pre-existing issue, fixing root cause" Get help: This is beyond simple restoration, need architectural fix
Sometimes "working" just meant "nobody noticed it was broken yet"
Logs Are Misleading or Corrupted
Scenario: Logs show "success" but payments are actually failing Don't trust logs blindly: Verify with actual user behavior data Cross-reference sources: Database records vs application logs vs user reports Check log pipeline: Is logging system itself broken? Manual verification: Test the actual user journey yourself Document discrepancy: "Logs unreliable during incident - used customer reports"
Logging systems fail too. Real user impact trumps what logs say
Backup System Is Worse Than Broken Primary
Scenario: Switch to backup payment processor but it has 50% failure rate Weigh the options: 0% success vs 50% success - 50% is still better Communicate trade-offs: "Backup has issues but better than nothing" Parallel work: Keep trying to fix primary while backup limps along Set expectations: Tell stakeholders backup is temporary bandaid Monitor closely: Backup might completely fail under load
Sometimes all your options are bad. Pick the least bad and keep working
Dependencies You Didn't Know About
Scenario: Fix payments but now user accounts are locked - turns out they're connected Map the blast radius: What else broke when you "fixed" the main issue? Emergency discovery: "What systems depend on payment service?" Prioritize fixes: Can users live with locked accounts short-term? Get architecture help: Someone needs to understand these connections Document discoveries: Update system diagrams with newly discovered dependencies
Complex systems have hidden connections. Your fix might break something else

HUMAN FACTORS

Incident Commander Is Panicking
Scenario: IC is new to role, clearly stressed, making contradictory decisions Stay calm yourself: Don't let their panic infect you Gently guide: "IC, should we focus on payments first or database?" Suggest structure: "Let's follow the runbook order" Escalate if needed: If IC is making dangerous decisions, get their manager Be the steady voice: Provide clear technical updates to anchor the team
Panic is contagious but so is calm. Be the steady presence the team needs
Expert Gives You Wrong Information
Scenario: Payment expert says "just restart the service" but that breaks it worse Trust but verify: "Let me check this in staging first" Question politely: "Are you sure? I'm seeing different behavior" Get second opinion: Ask another engineer to confirm the approach Document what happened: "Followed expert advice but it failed" Don't assume malice: Expert might be thinking of different system
Even experts make mistakes under pressure. Verify before acting on advice
Team Fighting During Incident
Scenario: Database team blaming application team while systems are down Don't engage in blame: "Let's fix first, discuss cause later" Focus on facts: "Database CPU is at 95%, payment API timing out" IC should mediate: "Everyone focus on their piece, blame game after resolution" Separate if needed: Put teams in different channels/calls Document neutrally: Focus on timeline and technical facts, not blame
Blame games waste time during incidents. Focus on resolution, accountability later
System Held Together with Duct Tape
Scenario: You discover payment system is a mess of hacks and workarounds Don't try to fix everything: Focus on immediate business impact only Work with the mess: Use existing hacks to restore service Document the horror: "System needs major refactoring after incident" Warn stakeholders: "This fix is temporary, system needs rebuilding" Plan for future: Start conversation about technical debt after incident
During incidents, work with what you have. Major refactoring comes later
Conflicting Information from Multiple Sources
Scenario: Monitoring says OK, users say broken, database team says overloaded Prioritize user reports: Real customer impact trumps monitoring Cross-check data sources: Find the source of truth Time-sync information: "Monitoring might be 5 minutes delayed" Make decisions with incomplete data: Can't wait for perfect information Communicate uncertainty: "Based on user reports, appears to be..."
During incidents, information is often contradictory. Use best available data

RESOURCE CONSTRAINTS

No Permissions to Fix What's Broken
Scenario: Need to restart database service but you don't have production access Don't waste time trying: You'll just get access denied errors Escalate permission request: "Need DB restart permissions for SEV1" Find someone who does: "Who has DB admin access right now?" Temporary workaround: Can you fix it differently with your permissions? Document the blocker: "5 minutes lost waiting for permissions"
Permission systems don't pause for incidents. Find alternatives or escalate fast
Vendor Support Takes Hours to Respond
Scenario: Stripe is down, their support ticket says "4 hour response time" Don't wait for vendor: Find workarounds immediately Check vendor status: Their public status page might have updates Use alternatives: Switch to different payment processor if possible Social media pressure: Sometimes @vendor on Twitter gets faster response Escalate to sales: Business account managers respond faster than support
Vendor SLAs don't match your business needs. Have backup plans
Expert Is Asleep in Different Timezone
Scenario: Payment architect is in Australia, it's 3AM there Calculate if it's worth it: How critical is this specific knowledge? Find their documentation: Check their recent notes, design docs Look for apprentice: Who has been learning from them? Try to solve without them: Use available resources first Wake them if critical: But have specific questions ready, not "system broken"
Global teams mean someone's always asleep. Don't depend on one person
Fix Requires Another Team That's Not Available
Scenario: Need network team to update firewall rules but they're all offline Find interim solution: Can you route around the network issue? Emergency escalation: Wake up network on-call if truly critical Business decision: Accept partial functionality vs waiting for proper fix Document dependency: "Resolution blocked by network team availability" Plan better coverage: Discuss 24/7 coverage for critical teams post-incident
Cross-team dependencies create bottlenecks. Have workarounds or escalation paths
Cloud Provider Has Regional Outage
Scenario: AWS us-east-1 is down and that's where your payment system lives Check AWS status: Confirm it's really AWS, not just you Failover if possible: Do you have multi-region setup? Communicate externally: "Experiencing issues due to AWS outage" Find workarounds: Can you process payments manually temporarily? Learn for next time: This exposes single points of failure
Cloud providers have outages too. Your incident response needs to handle dependencies failing

YOUR ACTUAL SCOPE AS APP SUPPORT

What You CAN'T Do (And That's Normal)
Reality check - as app support you typically DON'T have: No code deployments: Can't push fixes or rollback releases No database admin: Can't restart DB, change schemas, fix locks No infrastructure: Can't restart servers, change load balancer config No production secrets: Can't rotate API keys or change external configs Limited admin panels: Maybe read-only access to most systems
This isn't a limitation - it's proper separation of duties and security
What You CAN Do (More Than You Think)
Your actual powers during incidents: Application config: Change timeouts, retry counts, feature flags User management: Disable problematic accounts, reset user states Cache management: Clear application caches, refresh data API testing: Use Postman/tools to verify external services Data lookups: Query read-only databases to understand user impact
Application-level changes can often stop the bleeding without deployments
When You Hit Permission Walls
Scenario: You know the fix but can't execute it Don't waste time trying: "Permission denied" means move to next option Document the needed fix: "Need DB restart by admin to clear locks" Find someone who can: "Paging DBA on-call for database restart" Propose alternatives: "Can we disable the feature instead of fixing DB?" Time-box escalation: If no response in 10 minutes, try different approach
Permission boundaries are real. Work within them or escalate efficiently
Making the Most of Limited Access
Creative use of your actual permissions: Read-only database: Query to understand which users affected Application admin: Disable broken features via admin panel Log analysis: SSH to app servers and analyze application logs Configuration files: Change settings that don't require restart Monitoring tools: Create alerts and dashboards for the incident
Limited access doesn't mean powerless. Use what you have creatively
Your Value During Major Incidents
How app support engineers contribute to SEV1: Application expertise: You know the user flows and business logic best Investigation skills: You're good at log analysis and user impact assessment Communication bridge: Translate technical issues for customer support Coordination: Track what's been tried while engineers focus on fixes Documentation: Keep detailed timeline while others focus on resolution
You don't need admin access to be valuable. Your knowledge and skills matter most

CREATIVE WORKAROUNDS WITHIN YOUR SCOPE

Application-Level Configuration Changes
Scenario: Payment timeouts causing failures, no deployment access Increase timeout values: Change payment API timeout from 30s to 60s in config Reduce retry attempts: Change from 5 retries to 2 to reduce load Disable non-essential features: Turn off recommendation engine during payment Change batch sizes: Process payments in smaller groups to reduce DB load Enable debug logging: Increase log level to capture more detail
Configuration changes don't require deployments but can stop bleeding
Feature Toggles and Circuit Breakers
Scenario: New feature causing system overload Feature flag disable: Turn off the problematic feature via admin panel Circuit breaker enable: Activate circuit breaker for external API calls Partial rollout: Reduce feature from 100% users to 10% via toggles Maintenance mode: Enable "under maintenance" page for affected sections Rate limiting: Enable rate limiting for specific user actions
Modern apps have built-in controls you can use without code changes
User and Data Management
Scenario: Specific users or data causing system problems Disable problem accounts: Temporarily disable accounts causing high load Reset user sessions: Force logout for all users to clear corrupted sessions Clear user caches: Remove cached data for problematic users Block IP addresses: Use application firewall to block problematic traffic Data cleanup: Remove or archive old data that's causing queries to slow
Sometimes the problem is specific users or data, not the whole system
External Service Management
Scenario: Third-party service integration is failing Switch API endpoints: Change from primary to backup API URL Disable external calls: Turn off non-critical third-party integrations Change API keys: Rotate to backup API keys if primary is rate-limited Mock responses: Enable mock mode for external services during their outage Queue external calls: Change from real-time to queued processing
You often control how your app talks to external services
Communication and User Experience Fixes
Scenario: Can't fix the root cause but can improve user experience Update error messages: Change generic errors to specific helpful messages Enable maintenance banners: Warn users about known issues Redirect traffic: Send users to working pages instead of broken ones Enable offline mode: Switch app to cached/offline functionality Create status updates: Post detailed updates on status page for transparency
Sometimes you can't fix it but you can make the experience less frustrating

WHEN TO ESCALATE (AND TO WHOM)

Database Issues (Escalate to DBA Team)
Signs you need database admin help: Query performance: Slow queries affecting entire application Connection limits: "Too many connections" errors Lock issues: Queries timing out due to table locks Disk space: Database running out of storage space Replication lag: Read replicas falling behind master Escalate to: Database team, provide specific error messages and affected tables
Database problems require admin access you don't have
Infrastructure Problems (Escalate to DevOps/SRE)
Signs you need infrastructure help: Server resources: High CPU/memory across multiple servers Network issues: Connectivity problems between services Load balancer: Traffic routing problems Auto-scaling: Servers not scaling up during traffic spikes DNS issues: Domain name resolution problems Escalate to: DevOps/SRE team, provide affected servers and resource metrics
Infrastructure changes require system admin privileges
Code/Deployment Issues (Escalate to Development Team)
Signs you need code changes: Application bugs: Logic errors that require code fixes Deployment rollback: Need to revert to previous version Configuration deployment: Changes that require release pipeline Memory leaks: Application-level memory issues Performance optimization: Code that needs refactoring Escalate to: Development team, provide exact error reproduction steps
Code problems need people with deployment access and code knowledge
External Service Issues (Escalate to Vendor/Partnerships)
Signs you need vendor escalation: API rate limits: External service blocking your calls Service outages: Third-party provider having issues Contract issues: Need emergency support beyond normal SLA Integration problems: Complex API integration failing Performance issues: External service response times degraded Escalate to: Account manager or technical contact at vendor
External problems require relationship and contract leverage
How to Escalate Effectively
Best practices for escalation: Be specific: "Database locks on payment_transactions table" not "database slow" Provide context: Business impact, affected users, revenue loss Include evidence: Error messages, screenshots, log snippets Suggest urgency: "SEV1 - $50k/hour revenue impact" Offer to help: "I can provide more logs or test fixes" Set timeline: "Need response in 15 minutes for SEV1" Document everything: Keep timeline of who you escalated to and when
Good escalation gets faster response and better outcomes
